# 🏢 Корпоративная иерархия и Роли (Corporate Hierarchy)

Этот документ детально описывает модель подчинения и распределения ролей внутри клиентской организации.

## 1. Модель Оргструктуры

Система использует графовую модель для представления организации, где узлами являются подразделения (`OrgUnit`), а ребрами — связи подчинения.

### Типы подразделений (`UnitType`)
- `company`: Головная организация.
- `division`: Крупный блок (например, "Департамент ИТ").
- `department`: Отдел внутри дивизиона.
- `squad`: Команда или проектная группа.

## 2. Матрица Подчинения и Ролей

Каждый пользователь (`User`) связан с одним или несколькими подразделениями через таблицу `Membership`.

### Роли в элементе структуры
| Роля | Права | Описание |
| :--- | :--- | :--- |
| `Head` | `admin`, `full_access` | Руководитель подразделения. Видит всё внутри и ниже по иерархии. |
| `Lead` | `approve`, `write` | Лид направления или проекта. |
| `Contributor` | `propose`, `write` | Активный участник, может предлагать изменения (патчи). |
| `Observer` | `read_only` | Аудитор или заинтересованное лицо без права правки. |

## 3. Правила Наследования Прав

1.  **Сверху вниз:** Права, выданные на уровне `Division`, по умолчанию дают `read`-доступ ко всем дочерним `Department` и `Squad`.
2.  **Изоляция Workspace:** Доступ к конкретному рабочему пространству (`Workspace`) может быть ограничен только определенным `Squad`, даже если пользователь является `Lead` в соседнем подразделении.
3.  **Конфликт ролей:** Если пользователь имеет разные роли в разных узлах, система применяет наиболее приоритетную для конкретного контекста (Scope).

## 4. Пример структуры данных (JSON)

```json
{
  "unitId": "dept-it-001",
  "name": "Департамент Инфраструктуры",
  "parent": "division-tech",
  "members": [

      "userUid": "u-9923",
      "role": "Head",
      "since": "2025-01-01"
    }
  ],
  "workspaces": ["ws-core-vault", "ws-cloud-migration"]
}
```
