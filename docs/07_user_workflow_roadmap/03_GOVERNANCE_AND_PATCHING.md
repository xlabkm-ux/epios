# Жизненный цикл Governance и Patching

В основе Epistemic OS лежит парадигма: **графы решений не должны мутироваться бесконтрольно**. Любое значимое изменение (особенно меняющее архитектурный контекст) должно проходить через Governance Flow.

## Жизненный цикл ArtifactPatch

1. **PROPOSED (Предложен)**
   - Пользователь инициирует `ProposePatchUseCase`.
   - В БД (GovernanceRepository) создается запись `PatchGovernance` со статусом `proposed`.
   - Граф еще **не изменен**. Патч лишь хранит намерение (`targetNodeId`, `operation`, `content`).
   - UI отображает патч в блоке "Pending Review" внутри `GovernancePanel`.

2. **VOTING (Голосование)**
   - Ревьюеры применяют `CastVoteUseCase`.
   - Для сложных профилей может требоваться консенсус (несколько аппрувов). В MVP v1.1 достаточно одного голоса "Approve".

3. **APPROVED / REJECTED (Принят / Отклонен)**
   - Если голос положительный, патч переходит в `approved`.
   - Автоматически триггерится `ApplyPatchUseCase` (если это предусмотрено профилем).
   - `ApplyPatchUseCase` выполняет физическую мутацию графа (изменяет ноду, добавляет связь).

4. **FINALIZED (Финализирован)**
   - Система фиксирует новую версию `ReadinessAssessment`.
   - Генерируется новая версия `ArtifactVersion`.
   - Патч переходит в статус `finalized` (архивируется).

## Влияние на Workflow Пользователя

Такой подход защищает пользователя от "тихих" изменений:
- **Никаких сюрпризов:** Если Reviewer ушел на выходные, а Author добавил новые узлы, Reviewer увидит их как `Proposed Patches` по возвращении.
- **Traceability (Трассируемость):** При просмотре финального документа Observer может кликнуть на любое утверждение и увидеть, **каким патчем**, **кем** и **почему (Rationale)** оно было внесено в граф.

*В будущих итерациях планируется добавить возможность "Групповых патчей" (Batched Patches) — когда одно предложение изменяет сразу несколько узлов графа одновременно.*
