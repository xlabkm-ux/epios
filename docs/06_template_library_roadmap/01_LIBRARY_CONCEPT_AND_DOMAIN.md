# Концепция и Доменная модель (Template Library)

В MVP v1.1 Epistemic OS был сфокусирован на едином хардкодном сценарии: **ADR Review** с использованием статичного профиля `engineering-adr-review-v0.1`. 

Переход к **Template Library** означает превращение EpiOS в обобщенную платформу проверки адекватности (Adequacy Platform), где тип артефакта, правила его валидации и логика маппинга задаются динамически.

## 1. Ключевые понятия

### A. Artifact Template (Шаблон артефакта)
Описывает "форму" документа и его онтологию.
- **Структура:** JSON Schema или Markdown-шаблон, определяющий обязательные разделы (например, "Context", "Decision", "Consequences").
- **Онтология:** Какие типы узлов (Nodes) могут быть извлечены из этого текста (например, для ADR это `Claim` и `Evidence`, а для Threat Model это `Threat`, `Vulnerability`, `Mitigation`).
- **MCP Mapping:** Какой MCP-сервер или промпт должен использоваться для парсинга текста.

### B. Adequacy Profile (Профиль оценки / Readiness Profile)
Описывает "правила игры": как система решает, готов ли артефакт к утверждению.
- **Политики (Policies):** Набор правил (например, "Каждый `Threat` должен иметь связанный `Mitigation`").
- **Readiness Indicators:** Критерии готовности (например, "Охват доказательствами > 80%").
- **Hard Blocks:** Условия, при которых кнопка "Approve" жестко блокируется.

## 2. Развитие Доменной Модели

В рамках `packages/domain` произойдет расширение:
- Появится агрегат `TemplateRegistry` (или `OntologyRegistry`).
- Сущность `Mission` и `Workspace` будет ссылаться на `templateId` и `profileId` вместо хардкодной логики.
- Движок `ReadinessAssessment` (в application layer) станет "интерпретатором", который принимает граф и применяет к нему правила из `Adequacy Profile`, возвращая динамический статус.
