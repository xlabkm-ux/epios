Надо сделать не “еще одну ревизию документации”, а **карантин и классификацию**. Сейчас документация уже достаточно большая: я посмотрел архив — там **91 markdown-файл**, и только часть из них реально управляет ближайшим исполнением. Главная проблема: **64 файла не покрыты текущим `DOCUMENT_REGISTER.md`**, а значит для агента/разработчика они выглядят как “можно читать и исполнять”, хотя часть из них — roadmap, legacy или future scope.

Мой вердикт: **ничего не удалять сразу, но немедленно разделить документы на Authority / Contract / Delivery / Reference / Archive.**

---

## 1. Главный принцип

До стабильного ADR Review MVP:

```text
GitHub repo = source of truth.
Документы = контракты или справочные материалы.
Roadmap-файлы не являются задачами.
Future-файлы не являются обязательствами.
Legacy-файлы не являются инструкциями.
```

Это прямо согласуется с Governance Plan v1.3: до появления стабильного ADR Review MVP GitHub остается единственным источником истины, а любые governance-state изменения идут через PR, human review и CI.

## 5. Новые статусы документов

Ввести единый статус:

```text
accepted      — обязательный контракт
for_review    — можно читать, нельзя считать обязательным
reference     — справка, не instruction
deferred      — будущее, не MVP
superseded    — заменен другим документом
archived      — исторический материал
```

Запретить разнобой:

```text
Active
Accepted for MVP
Утвержден
Активен
Draft planning
```

Все это нормализовать в register.

## 6. Правило для агента и команды

В корень `docs/README.md` добавить:

```md
## Authority Rule

Only documents with status `accepted` in `DOCUMENT_REGISTER.md` are binding.

Documents with status `reference`, `deferred`, `archived` or `superseded`
must not be used as implementation instructions unless explicitly referenced
by an accepted ADR, GitHub Issue or PR.

Roadmap documents are not backlog.
Backlog lives in GitHub Issues.
```

Это критично. Иначе ИИ-агент будет хватать самый подробный roadmap и реализовывать не тот горизонт.

## 7. Что делать в ближайший PR

Создать один PR:

```text

```

Scope PR:

```text
1. Добавить Governance Plan v1.3 в docs/00_project/GOVERNANCE_PLAN.md.
2. Обновить docs/README.md под реальную структуру.
3. Обновить DOCUMENT_REGISTER.md: внести все 91 файла.
4. Пометить 01_DOC_GOVERNANCE.md как superseded.
5. Переместить ChatAVG файлы в 90_archive/chatavg_v2_4/.
6. Переместить future roadmap папки в 20_reference/.
7. Пометить API_CONTRACTS_MVP.md как for_review.
8. Оставить active только ADR Review MVP документы.
```

Definition of Done:

```text
- каждый .md файл имеет запись в register;
- нет активных ссылок на несуществующие папки;
- нет внешних ссылок на work_doc как authority;
- future roadmaps не имеют статус accepted/active;
- legacy docs не лежат рядом с active governance;
- агент может понять, что реализовывать сейчас.
```

С этой документацией надо сделать так:

```text
Не удалять.
Не развивать.
Не спорить дальше.

Классифицировать.
Зарегистрировать.
Разделить active / reference / archive.
Запретить roadmap как backlog.
Оставить authority только у accepted docs.
Все задачи перенести в GitHub Issues.
```

Сейчас документация — это не мусор. Это сырье. Но если не ввести статусность и authority rules, она станет источником хаоса. Your next move: **один PR на стабилизацию документации**, не новый концепт.
