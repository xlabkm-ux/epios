# План QA интерфейса и результаты

Обеспечение качества пользовательского интерфейса гарантирует, что визуальный wow-эффект не ломает функциональность, а ролевые модели работают безопасно.

## 1. План тестирования UI (Playwright E2E)

Для фронтенда выделен отдельный слой тестирования, который должен выполняться в CI/CD: `pnpm test:e2e`.

**Ключевые сценарии покрытия (Happy Paths):**
1. **Отображение Workspace:** Проверка, что при загрузке `demo-shell` корректно отрисовывается Canvas и все панели без ошибок в консоли.
2. **Асинхронный флоу:** Эмуляция SSE/Polling (мокирование API). UI должен корректно переходить от состояния "Loading Mapping" к отрисовке узлов.
3. **Patch Workflow:** 
   - Юзер кликает "Propose Patch".
   - Форма корректно принимает ввод.
   - После сабмита патч появляется в GovernancePanel.
4. **Approval Workflow:**
   - Юзер-ревьюер кликает "Approve".
   - ReadinessPanel меняет статус (например, с 🟡 на 🟢).
5. **Security UI:**
   - Авторизация под ролью `Viewer`.
   - Проверка, что кнопки "Vote" и "Propose Patch" отсутствуют в DOM или задизейблены (disabled).

## 2. Метрики Юзабилити (Target Metrics)

В процессе проведения пилотного тестирования (Pilot Phase) мы собираем следующие метрики:
- **TTC (Time To Complete):** Время, необходимое новому инженеру для прохождения полного флоу ревью ADR (Target: < 15 минут).
- **Error Rate:** Количество ошибочных кликов или обращений к модалкам "Помощь" (Target: 0 для опытных, < 3 для новичков).
- **System Trust Score:** Оценка (от 1 до 5), насколько пользователь доверяет статусу "Readiness", выданному системой (Target: >= 4/5).
