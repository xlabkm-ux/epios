# Модель Идентификации и Управления Доступом (Enterprise & Workspace RBAC)

Epistemic OS применяет гибридную ролевую модель, разделяющую контекст предприятия и контекст конкретного Рабочего места (РМ).

## 1. Контексты и Сущности

### А. Рабочее место (WorkPlace / WP / РМ)
Это первичный контекст пользователя, выбираемый в Настройках. РМ определяет "точку входа" и права доступа для текущей сессии.
- **Выбор:** Производится пользователем в окне "Настройка" из списка доступных ему назначений.
- **Таблица назначений:** Глобальный реестр, связывающий пользователя с его РМ.

### Б. Рабочее пространство (WorkSpace / WS / РП)
Это конкретный проектный или функциональный юнит (графы, документы), с которым пользователь работает внутри выбранного РМ.
- **Выбор:** Производится на "Слайдере" (в боковой панели / Sidebar).

### В. Таблица назначений (Table of Assignments)
Центральный реестр прав, доступный для редактирования только **Администраторам**. Каждая запись (строка) определяет:
- **Подразделение (Unit):** К какой части оргструктуры относится назначение.
- **Должность (Position):** Официальный статус пользователя в этом контексте.
- **Роль (Role):** Функциональные права (Owner, Reviewer, Contributor, Observer).
- **Рабочее пространство (WorkSpace):** К какому конкретно РП привязано это назначение.

## 2. Унифицированные Роли

| Уровень | Роль | Описание | Кто назначает |
| :--- | :--- | :--- | :--- |
| **Система** | **Administrator** | Полные права в системе: управление должностями, создание РМ (WP), ведение Таблицы назначений. | Система / Root |
| **WorkPlace** | **Owner** | Владелец РМ (WP). Управляет участниками своего WP и их ролями. | Administrator |
| **WorkPlace** | **Contributor** | Редактор. Может изменять данные, предлагать патчи и оценки внутри WP. | Owner / Admin |
| **WorkPlace** | **Reviewer** | Рецензент. Участвует в голосовании и утверждении изменений внутри WP. | Owner / Admin |
| **WorkPlace** | **Observer** | Наблюдатель. Только чтение данных внутри назначенного WP. | Owner / Admin |

## 3. Матрица Доступа (Access Control Matrix)

| Действие (Action) | Resource | Observer | Contributor | Reviewer | Owner | Admin |
| :--- | :--- | :---: | :---: | :---: | :---: | :---: |
| Просмотр данных РМ | `workspace.read` | ✅ | ✅ | ✅ | ✅ | ✅ |
| Инициация маппинга | `mapping.start` | ❌ | ✅ | ✅ | ✅ | ✅ |
| Оценка источника | `rating.create` | ❌ | ✅ | ✅ | ✅ | ✅ |
| Предложение патча | `patch.propose` | ❌ | ✅ | ✅ | ✅ | ✅ |
| Утверждение патча (Vote) | `patch.approve` | ❌ | ❌ | ✅ | ✅ | ✅ |
| Назначение ролей в РМ | `workspace.roles` | ❌ | ❌ | ❌ | ✅ | ✅ |
| Удаление/Архив РМ | `workspace.manage` | ❌ | ❌ | ❌ | ✅ | ✅ |
| Управление должностями | `user.position` | ❌ | ❌ | ❌ | ❌ | ✅ |
| Создание новых РМ | `workspace.create` | ❌ | ❌ | ❌ | ❌ | ✅ |

## 4. Архитектура контроля доступа

- **Semantic Integrity:** Агент отслеживает единообразие терминов "Должность" и "Роль" во всех документах и коде.
- **Switching Context:** При входе в систему пользователь выбирает одно из доступных ему РМ. Вся дальнейшая работа происходит в контексте выбранного РМ и соответствующей ему роли.
- **Domain Enforcement:** UseCases проверяют права на основе пары `(User, ActiveWorkspaceID)`.
