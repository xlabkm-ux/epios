# Жизненный цикл пользователя и Provisioning

В Enterprise-среде ручное создание пользователей (по одному) неэффективно. Требуется автоматизация управления учетными записями.

## 1. Стратегия Аутентификации (AuthN)

В перспективе (v1.2+) платформа откажется от хранения локальных паролей в пользу внешних Identity Providers (IdP).
- **Поддержка протоколов:** OAuth 2.0 / OpenID Connect (OIDC) и SAML 2.0.
- **Интеграции:** Microsoft Entra ID (Azure AD), Okta, Keycloak.
- **Схема работы:** EpiOS будет доверять проверку паролей/MFA внешнему провайдеру и принимать JWT токены, содержащие `email` и массив групп (`groups`).

## 2. Жизненный цикл (Provisioning Workflow)

### A. Onboarding (Регистрация / JIT Provisioning)
Когда пользователь впервые логинится через SSO:
1. Система проверяет наличие пользователя в таблице `users` по `email`.
2. Если пользователя нет, происходит Just-In-Time (JIT) регистрация.
3. Роль (`UserRole`) назначается автоматически на основе маппинга групп из IdP (например, группа `architects` -> роль `reviewer`, `all-staff` -> `observer`).

### B. Offboarding (Деактивация)
Удаление пользователя в системах Governance — опасная операция, так как сломает ссылочную целостность графа (AuditRecords и Patches ссылаются на `actorId`).
- **Soft Delete:** При получении сигнала от IdP или команды Admin, поле `isActive` переводится в `false`.
- Пользователь теряет доступ к API.
- В UI его аватарки и действия отображаются с пометкой `[Deactivated]`, но исторические решения (Approve, Propose) сохраняются навсегда.
