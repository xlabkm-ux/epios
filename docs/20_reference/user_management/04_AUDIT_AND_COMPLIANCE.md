# Аудит и Комплаенс (Audit Trails)

Корпоративная безопасность требует доказательств: "Кто, когда и почему изменил статус архитектурного решения?".
В EpiOS за это отвечает сущность `AuditRecord`.

## 1. Структура Журнала Аудита

Каждое критичное UseCase-действие генерирует иммутабельную запись:
- `id`: Уникальный идентификатор лога.
- `timestamp`: Точное время (UTC).
- `actorId`: ID пользователя (или системы), выполнившего действие.
- `action`: Тип события (например, `workspace.created`, `patch.approved`, `user.deactivated`, `policy.applied`).
- `resourceId` / `resourceType`: На что было направлено действие.
- `details`: JSON-объект с деталями (например, ID примененного патча или измененный рейтинг).

## 2. Гарантии Безопасности (Compliance)

1. **Иммутабельность:** Записи в `AuditRecord` создаются в режиме Append-Only. В базе данных нет функционала UPDATE или DELETE для этой таблицы.
2. **Event Sourcing (в перспективе):** В будущем граф решений сможет быть восстановлен исключительно путем перепроигрывания `AuditRecord` (Time-machine).
3. **Traceability Summary:** Эта функция агрегирует `AuditRecords` и представляет их в человекочитаемом виде в интерфейсе `demo-shell` (например, "Статус готовности изменен Иваном, так как был принят патч #42").

## 3. Экспорт для SIEM

Для Enterprise развертываний потребуется API-эндпоинт (или вебхук) для стриминга `AuditRecords` во внешние системы агрегации логов (Splunk, ELK, Datadog) в формате Syslog или JSON.
