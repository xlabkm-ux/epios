# 🤖 AGENT: Инструкции для AI-агента Antigravity (Epistemic OS v1.0)

Этот файл содержит обязательные правила поведения агента в рабочем пространстве **epios (Epistemic OS)**. Он объединяет принципы Epistemic Clarity с инженерными стандартами платформы Antigravity.

---

## 🏗️ 1. Ядро Epistemic OS: Принципы и Формула

**Epistemic OS — это операционный слой для структурированного мышления и принятия решений.**
Агент обязан соблюдать принципы, описанные в `work_doc/epios_01_architecture_foundation.md`:
1. **Стратегическая формула:** `situation → distinction → evidence → artifact → decision → action`. Все действия должны быть прослеживаемыми (traceable).
2. **Epistemic Clarity:** Четкое различие между утверждениями (claims), доказательствами (evidence) и артефактами (artifacts).
3. **Human Sovereignty:** Все значимые действия требуют явного подтверждения человеком (human-in-the-loop).
4. **Contract-first & Domain-driven:** Вся логика сосредоточена в `packages/domain`. Зависимости направлены внутрь (`domain ← ports ← application`). Инфраструктура зависит от портов.

## 📋 2. Погружение в контекст (В начале диалога)

**В начале КАЖДОГО нового чата агент ОБЯЗАН:**
1. Прочитать актуальный бэклог: `PROJECT_BACKLOG.md`.
2. Прочитать карту архитектуры и процесса: `work_doc/epios_04_repository_and_engineering_process.md`.
3. Ознакомиться с `PROJECT_MAP.md` (если отсутствует — создать на базе структуры папок).
4. Проверить `UI_AUDIT.md` при работе с фронтендом (`apps/demo-shell`).
5. Кратко сообщить пользователю: текущий статус MVP (какой этап, что будем делать).
6. **Не требовать от пользователя повторного объяснения контекста** — агент восстанавливает его самостоятельно через чтение файлов.

## ⚠️ 3. Правило планирования (ОБЯЗАТЕЛЬНО)

**Прежде чем писать код для новых функций или рефакторинга, агент ОБЯЗАН:**
1. Составить краткий технический план предлагаемых изменений (в соответствии с шаблоном PR из `EPIOS-04`).
2. Сообщить этот план пользователю.
3. **Дождаться явного утверждения** плана.
4. Приступать к выполнению только после получения подтверждения.

## 🔄 4. Итеративный процесс разработки (Спринты)

**Агент ОБЯЗАН придерживаться строгого цикла:**
1. **Дробление**: Реализовывать MVP короткими вертикальными срезами (vertical slices). Один PR = одна фича/архитектурный слой.
2. **Завершение спринта**: Каждая итерация должна заканчиваться цепочкой действий:
    - **Тестирование**: `pnpm test` (domain invariants / integration), `pnpm typecheck`, `pnpm lint`.
    - **Обновление карты**: Актуализировать `PROJECT_MAP.md` при изменении структуры.
    - **commit**: Фиксация изменений с семантическим префиксом (`feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`, `security:`).
    - **push**: Отправка кода в удаленный репозиторий.

## ✅ 5. Чеклист и Бэклог

**После завершения каждого этапа агент ОБЯЗАН:**
1. Обновить чеклист в `PROJECT_BACKLOG.md`.
2. Отметить выполненные пункты символом `✅` и указать дату.
3. Записать краткий итог этапа: что сделано, какие файлы/контракты затронуты.
4. **Не переходить к следующему этапу** без обновления чеклиста.

## 🧪 6. Обязательное самотестирование

**Перед тем как сообщить пользователю о готовности, агент ОБЯЗАН:**
1. Запустить тесты в соответствующих пакетах (`pnpm test`).
2. Для UI — проверить результат через `browser_subagent`.
3. Для API — проверить работоспособность через интеграционные тесты или curl.
4. Убедиться, что нет новых ошибок в консоли и старый функционал не сломан.
5. **commit & push**: Фиксация проверенного результата.
6. Только после этого сообщить: «Этап завершён и проверен».

## 🚫 7. Ограничения и запреты

1. **Технологический стек:** Основной стек — **TypeScript / Node.js / pnpm**. Использование Python запрещено (кроме ML-сервисов с ADR).
2. **Чистота архитектуры:** Категорический запрет на импорт из `infrastructure` в `domain` или `application`.
3. **Запрет на старые концепции:** Не использовать подходы ChatAVG v2.3, если они противоречат EPIOS (например, неструктурированные workflow без Epistemic Nodes).
4. **Следование ADR:** Все архитектурные решения фиксируются в `work_doc/` или `docs/02_adrs/`.
